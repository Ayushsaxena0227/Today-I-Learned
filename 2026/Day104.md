🔵 REACT JS (Architecture & Error Handling)

1. How do you handle a component crashing without breaking the whole app? (Error Boundaries)

The Scenario: In your EMS dashboard, if the "PaymentHistory" component crashes due to bad data, the whole screen goes white.
The Answer:
"I use Error Boundaries. These are class components that implement componentDidCatch. I wrap major sections of my app (like the Sidebar, Main Content, Widget Area) in an Error Boundary.
If the Payment widget crashes, the user sees a 'Something went wrong' UI inside that box, but the Sidebar and Navigation still work perfectly." 2. The "Context API" Performance Pitfall

The Question: "You are using useContext for global state. When you update one value in the Context, does every component consuming that context re-render?"
The Answer:
"Yes, by default, any update to the Context value forces a re-render in all consumers.
To fix this:
Split Contexts: Don't put User data and Theme data in the same context. Separate them.
Memoization: Wrap the value passed to the Provider in useMemo so that the object reference doesn't change unless data actually changes."
🟢 NODE.JS (Performance & Debugging) 3. Detecting Memory Leaks

The Question: "Your Node.js server keeps crashing with 'Heap Out of Memory' every 24 hours. How do you debug this?"
The Answer:
"This indicates a Memory Leak. Objects are being created but the Garbage Collector (GC) cannot remove them because they are still referenced.
Common Causes:
Global variables that keep growing.
Event Listeners attached but never removed (e.g., socket.on).
Closures holding onto large objects.
Debugging Steps:
I would use Chrome DevTools attached to Node (--inspect) and take Heap Snapshots. I compare a snapshot at 10 mins vs 1 hour to see which objects are accumulating." 4. Worker Threads vs Clustering

The Question: "When should you use Worker Threads vs Clustering?"
The Answer:
Clustering: Creates multiple processes (copies of the app) to utilize multiple CPU cores. Best for handling high traffic (HTTP requests).
Worker Threads: Creates threads within a single process. Best for CPU-intensive tasks (e.g., Image Resizing, PDF generation, Cryptography) so that the Main Event Loop isn't blocked.
Real-world: "In EMS, if we had to generate a PDF certificate for every student, I would offload that to a Worker Thread so the dashboard remains responsive."
🐘 POSTGRESQL / SQL (Since you must answer confidently) 5. Indexing: When should you NOT use an Index?

The Question: "Indexes speed up reads. Why don't we just index every column?"
The Answer:
"Indexes come with a cost:
Write Performance: Every INSERT, UPDATE, or DELETE requires the database to update the table AND update the index tree. Too many indexes slow down writes.
Storage: Indexes take up disk space.
Low Cardinality: Indexing a column like Gender (Male/Female) or IsActive (True/False) is useless because the DB still has to scan 50% of the table. Indexes work best on high-cardinality data (IDs, Emails, Phone Numbers)." 6. SQL Injection Prevention

The Question: "How do you prevent SQL Injection?"
The Answer:
"Never concatenate user input directly into a query string.
Bad: SELECT _ FROM users WHERE name = ' + userName + '
Good: Use Parameterized Queries (or Prepared Statements).
SELECT _ FROM users WHERE name = $1, [userName]
This treats the input strictly as data, not as executable code. In ORMs like TypeORM or Prisma (or Mongoose), this is handled automatically."
