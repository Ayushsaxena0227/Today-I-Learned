romise.all vs Promise.allSettled
The Question: "In your EMS Dashboard, you load User Profile, Course List, and Notifications simultaneously. If the 'Notifications' API fails, should the whole Dashboard crash? How do you handle this?"

The Answer:

Promise.all([...]): Fails fast. If one promise rejects, the entire thing rejects. The user sees an error screen. (Bad for dashboards).
Promise.allSettled([...]): Waits for all promises to finish, regardless of success or failure. It returns an array of objects: { status: 'fulfilled', value: ... } or { status: 'rejected', reason: ... }.
Resume Connection:
"In the EMS Dashboard, I used Promise.allSettled. If the 'Notifications' service is down, I still render the 'Courses' and 'Profile', and just show a small 'Failed to load' error in the notification bell area."

The Answer:

import { useState, useEffect } from 'react';

function useWindowSize() {
const [size, setSize] = useState({ width: window.innerWidth, height: window.innerHeight });

useEffect(() => {
const handleResize = () => {
setSize({ width: window.innerWidth, height: window.innerHeight });
};

    window.addEventListener('resize', handleResize);

    // CLEANUP FUNCTION (Very Important)
    return () => window.removeEventListener('resize', handleResize);

}, []);

return size;
}

The Question: "How do you prevent a hacker from hitting your Login API 1,000 times a second to guess passwords?"

The Answer:
"I implement Rate Limiting.
I use a middleware like express-rate-limit (or Redis for distributed systems).

If they exceed this, the server returns HTTP 429 Too Many Requests."

6. Where do you store JWTs? (Cookies vs LocalStorage)
   The Question: "In your Auth flow, where did you store the JWT token on the frontend?"

The Answer:

LocalStorage: Easy to implement, but vulnerable to XSS (if a hacker injects JS, they can read localStorage).
HttpOnly Cookie: More secure. JavaScript cannot read it, preventing XSS theft. However, it requires CSRF protection.
The "Safe" Answer:
"For my projects (EMS/Kartl), I stored them in LocalStorage for simplicity during the internship, but for a production banking app, I would use HttpOnly Cookies to prevent XSS attacks."

🗄️ DATABASE (MongoDB/SQL) 7. Compound Indexes
The Question: "In Kontiggo, users often search for Orders by status ('Paid') AND date (Last 30 days). If you index them separately, it's still slow. Why?"

The Answer:
"If I index status and date separately, MongoDB has to pick one index, find the matches, and then manually filter the rest.
The solution is a Compound Index:
db.orders.createIndex({ status: 1, createdAt: -1 })
This creates a single sorted list where orders are grouped by status, and within that status, they are sorted by date. The query becomes instant."

8. Soft Delete vs Hard Delete
   The Question: "When a user deletes a Course in EMS, do you actually remove the row from the database?"

The Answer:
"No, I prefer Soft Deletes.
Hard Delete: DELETE FROM courses WHERE id = 1 (Data is gone forever).
Soft Delete: I add a column isDeleted: boolean (or deletedAt: timestamp).
My queries then become: SELECT \* FROM courses WHERE isDeleted = false.
This allows me to restore data if a user made a mistake and keeps data for analytics."

🧠 SYSTEM DESIGN (Scaling) 9. Caching with Redis
The Question: "Your 'Get All Courses' API takes 200ms because it calculates average ratings from thousands of reviews. How do you make it faster?"

The Answer:
"I would implement Caching using Redis.

Check Cache: When the request comes in, check Redis for key all_courses.
Cache Hit: If found, return data immediately (2ms).
Cache Miss: If not found, query the Database (200ms), save the result to Redis with an expiration (TTL) of 10 minutes, and return data.
This reduces database load significantly."
