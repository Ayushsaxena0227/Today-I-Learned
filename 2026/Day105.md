REACT JS (Deep Dive Patterns)

1. Automatic Batching (React 18 vs Older)
   The Question: "If I call setCount(c => c + 1) three times inside a standard event handler, how many times does the component re-render? What if I do it inside a setTimeout?"

The Answer:
"In standard event handlers, React batches updates. So, calling setCount 3 times results in ONE re-render.

Before React 18: Updates inside setTimeout, Promises, or native DOM events were NOT batched (causing 3 re-renders).
In React 18: Automatic Batching covers everything. setTimeout, fetch, etc., all trigger only ONE re-render.
This improves performance significantly in complex apps like my EMS Dashboard." 2. Virtualization (Handling Large Lists)
The Question: "In your EMS Admin Panel, imagine you have a list of 10,000 students. Rendering them all at once freezes the browser. How do you fix this without pagination?"

The Answer:
"I would use Windowing (or Virtualization) using a library like react-window or react-virtualized.
Instead of rendering 10,000 DOM nodes, it only renders the items currently visible in the viewport (e.g., 20 items). As the user scrolls, it recycles the DOM nodes and swaps the data.
This keeps the memory footprint constant regardless of list size."

3. Custom Hooks vs HOCs
   The Question: "Why do we prefer Custom Hooks over Higher-Order Components (HOCs) nowadays?"

The Answer:
"HOCs (like withRouter or connect) often lead to Wrapper Hell—a deep component tree structure that is hard to debug. They can also cause naming collisions with props.
Custom Hooks allow us to share logic without adding extra components to the DOM tree. They are cleaner, easier to read, and easier to unit test."

🔒 NODE.JS & SECURITY (Crucial for your E-commerce projects) 4. XSS vs CSRF (Must Know)
The Question: "Explain Cross-Site Scripting (XSS) and Cross-Site Request Forgery (CSRF). How did you prevent them in Kartl?"

The Answer:

XSS (Cross-Site Scripting): Attacker injects malicious scripts (e.g., <script>stealCookies()</script>) into inputs (like a comment or product review).
Prevention: React escapes content by default. On the backend, I sanitize inputs using libraries like dompurify or validator before saving to DB.
CSRF (Cross-Site Request Forgery): Attacker tricks a logged-in user into making an unwanted action (like 'Delete Account') without them knowing.
Prevention: I used SameSite cookies (SameSite: Strict) and implemented CSRF Tokens (using csurf middleware) for sensitive state-changing requests. 5. What is a Buffer?
The Question: "You used Streams for file uploads. What is a Buffer in that context?"

The Answer:
"JavaScript (V8) wasn't designed to handle binary data (raw memory), it handles Strings.
A Buffer is a temporary chunk of memory used to hold binary data (like images, videos, or file streams) while it's being moved from one place to another (e.g., File System -> Network).
When I upload a file in Kontiggo, the data comes in chunks (Buffers). I collect these chunks and assemble them, or pipe them directly to storage."

🗄️ DATABASE DESIGN (Schema Modeling) 6. Polymorphic Associations (Advanced Modeling)
The Question: "In EMS, you have 'Comments'. A user can comment on a 'Course' OR a 'Mentor Profile'. How do you design the database schema for this?"

Option A (Bad): Two tables: CourseComments and MentorComments. (Hard to maintain).
Option B (The Interview Answer):
"I would use a Polymorphic Schema.
In SQL/Postgres, the Comments table would have:

id
content
entity_id (The ID of the course OR mentor)
entity_type (String: 'COURSE' or 'MENTOR')
In MongoDB, I can use a similar pattern or embed comments if the count is low. This allows me to query all comments for any entity efficiently." 7. SQL vs Mongo: Handling Product Variants (Kartl)
The Question: "In Kartl, a T-Shirt has Size (S, M, L) and Color (Red, Blue). A Laptop has RAM and Storage. How do you model these dynamic attributes in SQL vs MongoDB?"

The Answer:

SQL (Strict): Hard. You need an EAV (Entity-Attribute-Value) table pattern, which is slow and complex. Or use a JSONB column in Postgres (modern solution).
MongoDB (Flexible): Easy.
JSON

{
"productName": "T-Shirt",
"attributes": {
"size": ["S", "M"],
"color": ["Red"]
}
}
"This is why I chose NoSQL (Firebase/Mongo) for Kartl—e-commerce product attributes vary wildly, and a flexible schema prevents migration headaches."
📡 SYSTEM DESIGN (Real-World Scaling) 8. Scaling WebSockets (EMS Chat/Notifications)
The Question: "You implemented Notifications in EMS using Firebase/Sockets. If you have 1 Server, it works. If you scale to 5 Servers behind a Load Balancer, what breaks? How do you fix it?"

The Answer:
"WebSockets are stateful (persistent connections).
The Problem: User A is connected to Server 1. User B is connected to Server 2. If User A sends a message, Server 1 has it, but Server 2 (where User B is) knows nothing about it. User B never receives the message.
The Fix: Use Redis Pub/Sub (Publisher/Subscriber).

When Server 1 receives a message, it 'Publishes' it to a Redis channel.
All other Servers (2, 3, 4, 5) 'Subscribe' to that channel.
Redis acts as the central radio tower, broadcasting the message to all servers so they can push it to their connected users." 9. Designing a "Coupon Code" System (Concurrency)
The Question: "In Kartl, you have a promo code 'OFF50' that can only be used 100 times total. 1,000 users click 'Apply' at the same second. How do you prevent 1,000 users from getting the discount?"

The Answer:
"This is a classic Race Condition.

Atomic Decrement: I would NOT fetch the coupon, check count, and update. That's too slow.
Database Level: I use an atomic database operation.
MongoDB: db.coupons.findOneAndUpdate({ code: 'OFF50', usageCount: { $lt: 100 } }, { $inc: { usageCount: 1 } })
SQL: UPDATE coupons SET usage_count = usage_count + 1 WHERE code = 'OFF50' AND usage_count < 100;
Logic: If the database returns a modified document/row, the user gets the discount. If it returns null (or 0 rows affected), the limit is reached. The database handles the locking for me."
⚡ RAPID FIRE ROUND (Short, Confident Answers)
What is a CORS error?
Browser security preventing a frontend on Domain A from calling an API on Domain B. Fixed by setting Access-Control-Allow-Origin headers on the backend.
What is the difference between PUT and PATCH?
PUT replaces the entire resource. PATCH updates only specific fields.
What is a Closure in JS?
A function that remembers variables from its outer scope even after that outer function has finished executing.
Explain this keyword in Arrow Functions.
Arrow functions do not have their own this. They inherit this from the parent scope (Lexical Scoping).
What is Sharding in Database?
Splitting a large database into smaller, faster pieces (shards) across multiple servers to handle massive data.
