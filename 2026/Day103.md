ode.js Event Loop (Technical Definition)
The Interview Question: "Explain the internal architecture of Node.js. How does it handle concurrency?"

The Technical Answer:
"Node.js is a runtime environment built on Chrome's V8 JavaScript Engine. It uses an Event-Driven, Non-Blocking I/O model.

While JavaScript execution is indeed Single-Threaded (running on the Main Call Stack), the actual heavy lifting (File I/O, Network requests) is offloaded to Libuv.

Libuv is a C++ library that gives Node.js access to the Operating System's underlying asynchronous capabilities (like epoll in Linux). It maintains a Worker Pool (Thread Pool) for operations that the OS cannot handle asynchronously (like Crypto or File Compression).

The Workflow:

Call Stack: Executes synchronous code (variables, functions).
Web APIs / C++ Nodes: When an async function (like setTimeout or fs.readFile) is hit, it is offloaded here. The Main Thread continues running.
Callback Queue: Once the async operation finishes, its callback function is pushed here.
The Event Loop: This is an infinite loop that checks: "Is the Call Stack empty?"
If YES: It moves the first callback from the Queue to the Stack for execution.
If NO: It waits.
There are actually two queues:

Microtask Queue (Higher Priority): Promises (.then), process.nextTick.
Macrotask Queue (Lower Priority): setTimeout, setInterval, I/O.
Node processes all Microtasks before moving to Macrotasks." 2. Multi-Device Login & Session Logic (The "Trick")
The Scenario: You want to ensure a user can only be logged in on ONE device at a time.

Clarification on "Who gets rejected?":

The New Device (User trying to log in now): ✅ SUCCEEDS. They get into the account.
The Old Device (User logged in yesterday): ❌ GETS REJECTED. The next time they try to click a button, they get logged out.
How it works (Step-by-Step):

Database Setup: Add a field sessionId (string) to your User Schema.
Login Flow (Backend):
User sends Email/Pass.
You verify password.
CRITICAL STEP: Generate a unique string (e.g., uuid or timestamp). Let's say "Session_A".
Update the Database: Set user.sessionId = "Session_A".
Generate JWT: Include "sessionId": "Session_A" inside the token payload.
Send JWT to frontend.
The "Old Device" Problem:
The user logs in on a phone (Device B).
Backend generates "Session_B".
Updates Database to "Session_B".
Current State:
Database says: "Session_B"
Device A (Old Laptop) Token has: "Session_A"
Middleware Logic (The Rejector):
Device A tries to fetch profile.
Middleware decodes token -> sees "Session_A".
Middleware queries Database -> sees "Session_B".
Mismatch! ("Session_A" !== "Session_B").
Action: Return 401 Unauthorized. Frontend receives this and forcibly logs the user out.

The Interview Answer:
"The main difference is Inversion of Control (IoC).

Library (e.g., React, Lodash):

I am in control. I call the library functions when I need them.
I decide the folder structure, the routing, and the architecture.
Analogy: Buying furniture from IKEA. You pick the pieces, you build it how you want.
Framework (e.g., Angular, NestJS, Next.js):

The framework is in control. It calls my code.
It forces a specific structure (controllers, modules, services). If I put a file in the wrong place, the app won't run.
Analogy: Buying a model home. The structure, plumbing, and electricity are already decided; I just decorate the rooms."
