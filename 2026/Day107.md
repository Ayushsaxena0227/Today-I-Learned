1. Optimistic Concurrency Control (The "Versioning" Strategy)
   Context: In Kontiggo, imagine an Admin is updating a Product Price while a User is buying it. Or in EMS, two Admins try to edit the same Course description at the same time. Who wins?

The Question: "Two admins open the 'Edit Course' page at the same time. Admin A changes the title. Admin B changes the price. Admin A saves. Then Admin B saves. Admin A's changes are overwritten/lost. How do you prevent this?"

The "Senior" Answer:
"I use Optimistic Locking (Versioning).
In MongoDB, every document has a \_\_v (version key) field.

When Admin A and B fetch the document, they both get version: 1.
Admin A saves. The backend checks: does version == 1? Yes. Update happens, and version increments to 2.
Admin B tries to save. They send version: 1.
The backend checks: does version == 1? NO. The current version is 2.
The update is rejected, and Admin B is told: 'Someone else modified this record. Please refresh.'"
The Code (Mongoose):
Mongoose does this automatically if you handle the error, but here is the logic:

JavaScript

// Admin B tries to save
try {
const course = await Course.findOneAndUpdate(
{ \_id: courseId, **v: 1 }, // 🔥 ONLY update if version matches what I saw
{ $set: { price: 999 }, $inc: { **v: 1 } } // Increment version
);

if (!course) throw new Error("Data has changed! Please refresh.");
} catch (e) {
// Handle conflict

Context: In EMS, you send emails for course registration. In Kontiggo, you generate PDF invoices.

The Question: "If 1,000 users sign up for a Masterclass instantly, and your email service takes 1 second per email, your API will time out. How do you fix this?"

The Answer:
"I decouple the process using a Message Queue (like BullMQ with Redis, or RabbitMQ).

Producer: When a user signs up, the API just adds a 'job' to the Queue: { type: 'send-email', email: 'user@test.com' }. This takes 5ms. The API responds 'Success' immediately to the frontend.
Consumer (Worker): A separate background process reads from the Queue and sends the emails one by one (or in parallel batches).
If the email server crashes, the job stays in the Queue and retries later. The user never experiences lag."
Visual Explanation:

Without Queue: User -> API -> (Wait 5s) -> Email Sent -> Response.
With Queue: User -> API -> Add to Queue -> Response (Instant). (Worker handles email separately). 3. Server-Sent Events (SSE) vs WebSockets
Context: In Kontiggo, you want to show "Order Status: Processing -> Shipped".

The Question: "For an order tracking page, would you use WebSockets or Server-Sent Events (SSE)? Why?"

The Answer:
"I would use Server-Sent Events (SSE).

WebSockets are bi-directional (Chat apps). They are complex (need handshakes, heartbeat, reconnection logic).
SSE is mono-directional (Server -> Client). It uses standard HTTP.
Since order status updates only go from Server to User, SSE is lighter, easier to implement, and works over standard HTTP/2 without complex socket servers."
The Code (Backend - Express):

JavaScript

app.get('/order-updates', (req, res) => {
// Keep connection open
res.setHeader('Content-Type', 'text/event-stream');
res.setHeader('Cache-Control', 'no-cache');
res.setHeader('Connection', 'keep-alive');

// Send data periodically
const interval = setInterval(() => {
res.write(`data: ${JSON.stringify({ status: 'Shipped' })}\n\n`);
}, 5000);

// Cleanup when client closes tab
req.on('close', () => clearInterval(interval));

Context: You used NestJS in EMS/Kontiggo. This is a framework-specific but crucial concept.

The Question: "You listed NestJS. What is Dependency Injection (DI) and why is it better than just importing files?"

The Answer:
"Dependency Injection is a design pattern where the framework provides the objects (dependencies) a class needs, rather than the class creating them itself.

Without DI (Bad): Inside UserController, I write const paymentService = new PaymentService();. This makes testing hard because I can't swap the real payment service for a fake one.
With DI (NestJS): I define constructor(private paymentService: PaymentService) {}. NestJS automatically gives me the instance.

Context: EMS has a search bar for Courses.

The Question: "You want to search for courses that contain the word 'Javascript'. A standard index doesn't work well. Why?"

The Answer:
"A standard index (B-Tree) is good for exact matches (price = 100) or ranges (price > 100). It is bad for searching inside strings.
For search, I use a Text Index (In MongoDB: $text).
It breaks the string into tokens ('Intro', 'to', 'Javascript').
